Known Subclasses:
job.Job
Tasks represent asynchronous API calls. They are only created by invoking an asynchronous method (Tasks should not be created directly) on a SAGA object which returns a Task object (with saga.Task.Async or saga.Task.Task). But as saga.job.Job instances inherit from the Task class, Jobs are also effectively created as Tasks. If a Task gets created, it will share the state of the object it was created from.

Precondition: Task is in 'NEW' state

Postcondition: Task is in 'RUNNING' state

Permission: appropriate permissions for the method represented by the Task

Notes:
  • run can only be called on a Task in 'NEW' state. All other states will cause the 'IncorrectState' exception to be raised.
  • a 'Timeout' or 'NoSuccess' exception indicates that the backend was not able to start the Task.

cancel(self, timeout=0.0)

 
Cancel the asynchronous operation.
Parameters:
  • timeout (float) - time for freeing resources
Raises:
  • NotImplemented -
  • IncorrectState -
  • Timeout -
  • NoSuccess -

Precondition: Task is in 'RUNNING' state.

Postcondition: Task is in 'CANCELED' state.

Notes:
  • for resource deallocation semantics and timeout semantics, see Section 2 of the GFD-R-P.90 document.
  • if cancel() fails to cancel the Task immediately, and tries to continue to cancel the Task in the background, the Task state remains 'RUNNING' until the cancel operation succeeded. The state then changes to 'CANCELED'.
  • if the Task is in a final state, the call has no effect, and, in particular, does NOT change the state from 'DONE' to 'CANCELED', or from 'FAILED' to 'CANCLED'. This is to avoid race conditions.
  • if the Task is in 'NEW' state, an 'IncorrectState' exception is raised.
  • a 'NoSuccess' exception indicates that the backend was not able to initiate the cancelation for the Task.

wait(self, timeout=-1.0)

 
Wait for the Task to finish.
Parameters:
  • timeout (float) - seconds to wait
Returns: bool
indicator if the Task is done running
Raises:
  • NotImplemented -
  • IncorrectState -
  • Timeout -
  • NoSuccess -

Precondition: Task is not in 'NEW' state.

Postcondition: if no timeout occurs, Task is in a final state.

Notes:
  • wait returns success (True) as soon as the Task enters a final state
  • if the Task is already in a final state, the call returns success (True) immediately.
  • if the Task is in 'NEW' state, an 'IncorrectState' exception is raised.
  • wait returns no success (False) if the Task is, even after timeout, not in a final state.
  • a 'Timeout' or 'NoSuccess' exception indicates that the backend was not able to wait for the Task. Note that a 'Timeout' exception does not indicate that the Task is not in a final state after the given wait period - that causes an unsuccessfull (False) return value.
  • for timeout semantics, see Section 2 of the GFD-R-P.90 document

rethrow(self)

 
Re-raise any exception a failed Task caught.
Raises:
  • NotImplemented -
  • IncorrectURL -
  • BadParameter -
  • AlreadyExists -
  • DoesNotExist -
  • IncorrectState -
  • PermissionDenied -
  • AuthorizationFailed -
  • AuthenticationFailed -
  • Timeout -
  • NoSuccess -
Notes:
  • this method does nothing unless the Task is in 'FAILED' state, and also raises 'IncorrectState' if the Task is in any other state.
  • if in 'FAILED' state, the method raises an exception which indicates the reason why that Task entered the 'FAILED' state (i.e. it raises the exception which caused it to enter the 'FAILED' state.
  • rethrow() can be called multiple times, always raising the same exception.
